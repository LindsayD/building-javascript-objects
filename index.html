Objects
  • An Object:
  • Represents an instance of a "thing"
  • Stores state (adjectives) about the thing
  • Provides actions (verbs) to interact or change state
  • != a Class

  • Object Literals are:
  • Fundamental even to other patterns
  • Easy to create
  • Easy to extend instances
  • Cumbersome to create many of the same type
  • Cumbersome to extend many in the same way
  • Unable to keep data private
  • Unable to indicate an API
  • Can introduce namespace conflicts
  • Can cause problems with pointers vs values

  • Object Extension
  • Facilitates composition of functionality (Mixins)
  • Facilitates composition of data
  • Provides means to specify defaults
  • Still cumbersome to create many of the same type
  • Still no way to keep data private
  • Provides composition API definition*
Module Constructors
var person1 = personFactory(person1Data);
var personFactory = (function(personData) { var personDefaults = { ... }; function getHouse() { ... } function brandish() { ... } function scheme() { ... } var person = _.extend({ getHouse: getHouse, brandish: brandish, scheme: scheme }, personDefaults, personData }); return person; // composed object: data, defaults, methods )();
var personFactory = (function(personData) { var personDefaults = { ... }; var isDeadYet = false; /* functions go here */ return _.extend({ /* public functions ... */ isDeadYet: isDeadYet }, personDefaults, personData }); // composed object: data, defaults, methods )();
var personFactory = (function(personData) { var personDefaults = { ... }; var isDeadYet = false; /* functions go here */ return _.extend({ /* public functions ... */ isDeadYet: function(val) { return isDeadYet; }; }, personDefaults, personData }); // composed object: data, defaults, methods )();
var personFactory = (function(personData) { var personDefaults = { ... }; var isDeadYet = false; /* functions go here */ return _.extend({ /* public functions ... */ isDeadYet: function(val) { if (val) isDeadYet = val; return isDeadYet }; }, personDefaults, personData }); // composed object: data, defaults, methods )();
  • Module Pattern Construction
  • Still facilitates composition of functionality (Mixins)
  • Still facilitates composition of data
  • Still provides means to specify defaults
  • Makes it easy to create many of the same type
  • Gives ability to keep data private
  • Provides composition API and "intentional" API*
  • May consume more memory because a closure is created for each construction.
Prototype Inheritance
Favor object composition over class inheritance.
- the Gang of Four, Design Patterns
  • Prototype Inheritance
  • Provides instant extension of instances
  • Still facilitates composition of data
  • Still provides means to specify defaults
  • Makes it easy to create many of the same type
  • Unable to keep data private
  • Provides a concrete API (and "type" checking)
  • Cannot protect from API modification at runtime
  • Hierachies can get ugly
  • Saves memory because the prototype defined once
