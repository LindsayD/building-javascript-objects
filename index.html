Objects
  • An Object:
  • Represents an instance of a "thing"
  • Stores state (adjectives) about the thing
  • Provides actions (verbs) to interact or change state
  • != a Class

  • Module Pattern Construction Good Stuff:
  • Defines a public API for constructed objects
  • Provides means to specify defaults
  • Makes it easy to create many of the same type
  • Gives ability to keep data/functionality private
  • Module Pattern Construction Bad Stuff:
  • Does not provide "type" checking
  • No automatic instance extension
  • Closure for each object creation may consume more memory than alternatives
Prototype Inheritance
Favor object composition over class inheritance.
- the Gang of Four, Design Patterns
The Constructor
function Person(init) { this.name = init.name || this.name; this.weapon = init.weapon || this.weapon; this.house = init.house || this.house; }
The Instance
var person1 = new Person(person1Data);
The Prototype - Functionality
Person.prototype.getHouse = function() { return this.house; }; Person.prototype.brandish = function() { alert(this.name + ' brings forth ' + this.weapon); }; Person.prototype.scheme = function() { ... }; Person.prototype.die = function() { this.isDeadYet = true; };
The Prototype - Data Defaults
var personDefaults = { isDeadYet: false, gender: 'female', location: "King's Landing", house: 'Lannister' }; _.extend(Person.prototype, personDefaults); // extension works!
The Prototype - Instant and Automatic
var personDefaults = { isDeadYet: false, gender: 'female', location: "King's Landing", house: 'Lannister' }; _.extend(Person.prototype, personDefaults); // extension works!
var person1 = new Person({ name: 'Sansa' }); Person.prototype.favoriteFood = 'lemon cakes'; alert(person1.name + ' likes ' + person1.favoriteFood); // says "Sansa likes lemon cakes"
Instance vs Prototype
var person1 = new Person({ name: 'Sansa' }); Person.prototype.favoriteFood = 'lemon cakes'; alert(person1.name + ' likes ' + person1.favoriteFood); // says "Sansa likes lemon cakes" var person2 = new Person({ name: 'The Hound' }); person2.favoriteFood = 'chicken'; alert(person2.name + ' likes ' + person2.favoriteFood); // says "The Hound likes chicken"
Instance vs Prototype
var person1 = new Person({ name: 'Sansa' }); Person.prototype.favoriteFood = 'lemon cakes'; alert(person1.name + ' likes ' + person1.favoriteFood); // says "Sansa likes lemon cakes" var person2 = new Person({ name: 'The Hound' }); person2.favoriteFood = 'chicken'; alert(person2.name + ' likes ' + person2.favoriteFood); // says "The Hound likes chicken" alert(person2.constructor.prototype.favoriteFood); // says "lemon cakes"
FINISH!! Inheritance
var person = new Person({ name: 'Sansa' }); function Female(data) { ... }; Female.prototype = person; var female = new Female(data); function Queen(data) { ... }; Queen.prototype = female;
FINISH!!! Prototype Chaining
var person1 = new Person({ name: 'Sansa' }); Person.prototype.favoriteFood = 'lemon cakes'; alert(person1.name + ' likes ' + person1.favoriteFood); // says "Sansa likes lemon cakes" var person2 = new Person({ name: 'The Hound' }); person2.favoriteFood = 'chicken'; alert(person2.name + ' likes ' + person2.favoriteFood); // says "The Hound likes chicken" alert(person2.constructor.prototype.favoriteFood); // says "lemon cakes"
  • Prototype Inheritance Good Stuff:
  • Provides instant extension of instances
  • Facilitates composition of data
  • Provides means to specify defaults
  • Makes it easy to create many of the same type
  • Provides a concrete API (and "type" checking)
  • Saves memory because the prototype defined once
  • Prototype Inheritance Bad Stuff:
  • Object extension can't be used directly on this
  • Prototype extension can cause side effects
  • Unable to keep data private
  • Cannot protect from API modification at runtime
  • Deep hierachies can get ugly
Which do you use?
  • Use Object Literals
  • As quick data constructs
  • For "throw away" objects
  • Use Object Extension to
  • Combine data with defaults
  • Compose objects from multiple data sources
  • Compose components with Mixins
  • Facilitate other forms of object construction
  • Use the Module Pattern to
  • Hide internal implementation
  • Create objects with a steady API when you don't need inheritance
  • Avoid polluting the global scope
  • Implement dependency injection (testing)
  • Implement dependency resolution (dynamic loading)
  • Facilitate other forms of object construction
  • Use Prototype Inheritance to
  • Create an inheritance hierarchy*
  • Feel a bit more like working with classes**
  • Specify default data and behavior
  • Save memory by prototype efficiency
  • Extend objects you might not have created***
  • Implement other forms of object construction
